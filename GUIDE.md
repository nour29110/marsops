# MarsOps 

A plain-English breakdown of how the system works, with the math explained simply.

---

## 1. What is MarsOps?

An autonomous Mars rover mission planner. Given a terrain map, it:

1. Loads a map of Jezero Crater (synthetic math or real USGS satellite data)
2. Plans a route using A* pathfinding
3. Tests the route via a silent "dry-run" simulation
4. Executes the mission step-by-step, tracking battery/position/heading
5. Handles mid-mission problems (dust storms, stuck wheels, overheating)
6. Records everything as telemetry and generates a Markdown report
7. Exposes all of this through an MCP server (Claude Desktop) and a REST/WebSocket API (web UI)

## 4. The Terrain Layer

### What "terrain" means in code

The Mars map is a **2D grid of elevation numbers** (a Digital Elevation Model / DEM).
Every cell = one height value in meters. Jezero Crater floor sits around **-2600 m**
below the Mars areoid (the Martian equivalent of sea level).

Stored as a NumPy `float32` array:

```
row 0: [-2601.2, -2603.4, -2598.1, ...]
row 1: [-2605.7, -2607.0, -2600.3, ...]
...
```

### Synthetic vs. Real terrain

**Synthetic (default):** Generated by math — no internet needed, always identical (seed 42).

Components layered together:

| Layer | Formula | Effect |
|-------|---------|--------|
| Large hills | `40 × sin(2π×r/150) × cos(2π×c/150)` | Gentle rolling terrain |
| Medium bumps | Same but rotated 30°, amplitude 15 m | Breaks grid alignment |
| Small noise | `Normal(0, 3)` with seed 42 | Realistic roughness |
| Crater | `-60 × exp(-r² / (2 × 100²))` | Bowl depression in center |
| Delta ramp | `30 × max(0, 1 - 2r/rows) × max(0, 1 - 2c/cols)` | NW elevation ramp |
| Base offset | `-2600` | Shifts to Jezero floor elevation |

**Real (optional):** Downloads a 9 MB GeoTIFF from USGS Astrogeology. If it fails,
it raises an error — it does **not** silently fall back to synthetic (fail loudly).

### Slope calculation

Uses a **3×3 neighbourhood gradient** around each cell:

```
slope_deg = degrees( arctan( sqrt(gx² + gy²) ) )
```

Where `gx` and `gy` are the horizontal and vertical gradients computed by NumPy's
`np.gradient()` over the 3×3 window. Edge cells (no full window) return `0.0`.

### Traversability rule

A cell is driveable if **all three** are true:
1. Within grid bounds
2. Not a nodata sentinel (`-9999.0` means no satellite reading)
3. `slope_deg ≤ 25°` (real Curiosity rover operational limit, per JPL)

### Degrees → Meters conversion

The real DEM uses geographic coordinates (degrees). The code converts to meters using
Mars's mean radius:

```
π × 3,389,500 m / 180° ≈ 59,274 m per degree of arc
```

### Downsampling

500×500 grid → too slow for A*. Default: **5× downsample** → 100×100 grid.
Method: `elevation[::5, ::5]` (keep every 5th pixel). Resolution metadata updated accordingly.

## 5. The Pathfinding Layer (A*)

### What A* solves

Find the **cheapest path** between two cells on the grid without exploring every
possible route.

### The two scores every cell gets

- **g-score** = actual cost paid to reach this cell from the start
- **h-score** = estimated (heuristic) remaining cost to the goal
- **f-score** = `g + h` — A* always expands the cell with the lowest f-score

### 8-connected movement

The rover moves like a chess king — 8 directions. Move costs:

```
Straight move (up/down/left/right):  cost = 1.0 × cell_cost
Diagonal move:                        cost = √2 × cell_cost  (≈ 1.414)
```

Why √2? Because diagonals are longer: `√(1² + 1²) = √2`

### The heuristic — Octile Distance

```
h = dx + dy + (√2 - 2) × min(dx, dy)
```

Where `dx = |goal_col - col|` and `dy = |goal_row - row|`.

This is **admissible** — it never overestimates the true cost.
Admissibility guarantees A* always finds the optimal path.

> If the heuristic overestimates, A* might skip the real best path because it
> thinks another looks cheaper. Admissible = always correct.

### The Cost Function — Slope-Squared

```
cell_cost = 1.0 + (slope_deg / 10.0)²
```

| Slope | Cost |
|-------|------|
| 0° (flat) | 1.0 |
| 10° | 2.0 |
| 20° | 5.0 |
| 25° (rover limit) | 7.25 |

The **quadratic growth** is the key design choice. A path twice as long but flat
often beats a short steep path. This biases routes toward flat ground without making
steep cells completely forbidden.

Inspired by JPL's **GESTALT** hazard-assessment system used on real Mars rovers.

### Stale entry handling

A* uses a min-heap (priority queue). When a cheaper path to a cell is found, the
old entry isn't removed — it's left in the heap. When it comes up, it's skipped:

```python
if g_current > g_score.get(current, math.inf):
    continue  # stale entry, skip
```

This is "lazy deletion" — cheaper than removing from the heap.

### One-sentence summary

> A* finds the lowest-cost path across the terrain grid using a slope-squared cost
> function — flat cells cost 1.0, steep cells cost up to 7× more — so the planner
> naturally routes around hills without needing explicit obstacle rules.

## 6. The Mission Planner (`mission_planner_runtime.py`)

### What it does

Takes a natural-language description + start position and produces a validated,
energy-feasible `MissionPlan`. Four steps:

### Step 1 — Candidate waypoint generation

Samples a grid of evenly-spaced candidate points in the region of interest.
Filters to traversable cells only. Count = `min_waypoints × 3` (oversample, then filter down).

### Step 2 — Keyword filtering

Scans the mission description for four keywords and applies terrain filters:

| Keyword | Filter applied |
|---------|---------------|
| `flat` | `slope < 10°` |
| `high` | `elevation > median + 1 std_dev` |
| `low` | `elevation < median - 1 std_dev` |
| `delta` | `elevation < median` |

If keyword filtering leaves too few candidates, falls back to all traversable points.

### Step 3 — Nearest-Neighbour TSP ordering

Orders waypoints greedily: always pick the closest unvisited candidate next.

```
current = start
for each waypoint to select:
    pick the candidate closest to current
    current = that candidate
```

Not optimal (true TSP is NP-hard) but fast and good enough for small waypoint counts.

### Step 4 — Dry-run refinement loop (up to 5 iterations)

```
for i in range(5):
    simulate the full path cheaply (no telemetry)
    if battery_remaining >= 20% and duration ok:
        plan is feasible → stop
    else:
        drop the waypoint farthest from start
        retry
```

If still infeasible after 5 drops → return plan with `feasible=False`.

### One-sentence summary

> The planner samples candidate waypoints, filters them by terrain keywords,
> orders them with a nearest-neighbour heuristic, then iteratively drops the
> farthest waypoint until the dry-run simulation confirms the mission is
> energy-safe.

## 7. The Rover Physics Model (`simulator/rover.py`)

### RoverConfig defaults (Perseverance-class)

| Parameter | Value | Meaning |
|-----------|-------|---------|
| `battery_capacity_wh` | 2000 Wh | Total usable battery |
| `idle_draw_w` | 50 W | Power draw while stopped |
| `drive_draw_w` | 120 W | Power draw while driving |
| `speed_mps` | 0.042 m/s | Max drive speed (real Perseverance limit) |
| `drive_efficiency` | 0.5 | Rover only moves 50% of travel time (stops for imaging/comms) |
| `low_battery_threshold_pct` | 20% | Triggers low-battery event |

### Energy consumed per step

```
dist_m      = √(dr² + dc²) × resolution_m_per_cell
travel_s    = dist_m / speed_mps
energy_wh   = drive_draw_w × travel_s / 3600 × drive_efficiency
```

### Why battery always ends ~40-50% on the demo grid

- Grid: 100×100 cells after 5× downsample
- Cell size: ~296 m each
- Typical mission path: ~15-20 cells
- Math: 15 cells × 296 m = 4440 m → 4440 / 0.042 = 105,714 s → × 120W / 3600 × 0.5 = **1763 Wh used** from 2000 Wh → ~12% remaining

That's actually tight — but the planner ensures ≥ 20%, so it trims waypoints
until the route is short enough. Result: plans consistently land in the 40-50% range.

### Battery percentage

```
battery_pct = 100.0 × battery_wh / battery_capacity_wh   (clamped 0–100)
```

### Heading calculation

```
heading_deg = degrees( atan2(delta_east, -delta_south) ) % 360
```

Grid convention: row increases downward (south), col increases rightward (east).
0° = north, clockwise. Standard compass bearing.

## 7b. Rover Physics — Step-by-Step Math

### Per-step energy calculation

```
dist_m    = √(dr² + dc²) × resolution_m_per_cell
travel_s  = dist_m / speed_mps
energy_wh = drive_draw_w × travel_s / 3600 × drive_efficiency
```

Example — one straight cell on the demo grid (296 m/cell):
```
dist_m    = 1 × 296 = 296 m
travel_s  = 296 / 0.042 ≈ 7,047 s  (~2 hours per cell!)
energy_wh = 120W × 7047 / 3600 × 0.5 ≈ 117 Wh
```

### Heading calculation

```
heading_deg = degrees( atan2(delta_east, -delta_south) ) % 360
```

Grid convention: row increases downward (south), col increases rightward (east).
The `-delta_south` flips the axis so 0° = north, clockwise. Standard compass bearing.

### Execution event sequence

```
mission_start       ← always first
step                ← one per cell move
waypoint_reached    ← when rover lands on a planned waypoint
low_battery         ← exactly once, first time battery < 20%
anomaly             ← when an injected anomaly fires
recovery_replan     ← when recovery replanning succeeds
mission_complete    ← rover reaches the end
mission_failed      ← battery exhaustion or abort
```

Key design: `low_battery` fires **exactly once** (boolean flag). The engine **never raises** — all failures are caught and recorded as `mission_failed` events.

### Anomaly effect formulas

| Anomaly | Battery drain | Idle time added |
|---------|--------------|----------------|
| `dust_storm` severity s | `15 × s` % of total capacity | `3600 × s` seconds |
| `wheel_stuck` any s | None | None (blocks cells only) |
| `thermal_alert` severity s | None (but idle draws power) | `7200 × s` seconds |

Idle drain: `idle_draw_w × forced_idle_s / 3600 = 50W × idle_s / 3600`

### Recovery priority order

```
1. battery < 10%      → abort_to_start immediately
2. replan_around      → keep all non-blocked waypoints, replan from current position
3. reduce_ambition    → drop farthest waypoint, retry until feasible
4. abort_to_start     → nothing works → drive home via A*, emit mission_failed
```

Max 5 recovery attempts. Abort drive is best-effort — fails silently if A* can't find home route.

## 8. Anomalies and Recovery

### Three anomaly types

| Type | Effect |
|------|--------|
| `dust_storm` | Drains `15 × severity`% battery + forces `3600 × severity` seconds idle |
| `wheel_stuck` | Marks a set of cells as impassable (triggers replanning) |
| `thermal_alert` | Forces `7200 × severity` seconds idle (rover must cool down) |

### Recovery priority order

```
1. battery < 10%          → abort_to_start immediately (too dangerous)
2. replan_around          → replan from current position, keep non-blocked waypoints
3. reduce_ambition        → drop farthest waypoints one by one until plan is feasible
4. abort_to_start         → no combination works → navigate back to start via A*
```

Max 5 recovery attempts per mission to prevent infinite replanning loops.

## 11. Recovery Decision Logic

### Priority order

```
1. battery < 10%      → abort_to_start immediately
2. replan_around      → keep all non-blocked waypoints, replan from current position
3. reduce_ambition    → drop farthest waypoint one at a time until feasible
4. abort_to_start     → nothing worked → navigate home via A*, emit mission_failed
```

### Key design decisions

- Euclidean distance used for "farthest waypoint" — fast approximation, not A* path distance
- `must_return_to_start = False` forced during recovery — too energy-expensive mid-emergency
- `recover_from_anomaly()` never raises — all exceptions caught, demoted to `abort_to_start`
- Max 5 recovery attempts per mission — prevents infinite replanning loops
- Abort drive is best-effort — fails silently if A* can't reach home

## 12. WebSocket Telemetry Replay Flow

```
POST /api/command (execute_mission)
  → mission runs in worker thread via asyncio.to_thread (doesn't block event loop)
  → after completion: replay each TelemetryEvent through broadcaster (100ms delay default)
  → all WebSocket subscribers receive events in order → animates rover in web UI
```

Broadcaster is pub/sub: `subscribe()` returns an `asyncio.Queue`,
`broadcast(payload)` pushes to every subscribed queue.

## 13. Testing Strategy

### Two testing approaches

**pytest** — standard unit and integration tests. Covers every module:
`test_astar`, `test_rover`, `test_engine`, `test_recovery`, `test_mission_planner_runtime`, etc.

**Hypothesis** — property-based testing. Instead of writing specific test cases,
you describe *properties* that should always hold and Hypothesis generates hundreds
of random valid inputs to try to break them. Example:

```python
# Property: A* path always starts at start and ends at goal
@given(valid_terrain, valid_start, valid_goal)
def test_astar_endpoints(terrain, start, goal):
    path = astar(terrain, start, goal)
    assert path[0] == start
    assert path[-1] == goal
```

### Coverage setup

```toml
addopts = "-ra --cov=src/marsops --cov-report=term-missing --timeout=30"
```

30-second timeout per test — prevents infinite A* loops from hanging CI.
